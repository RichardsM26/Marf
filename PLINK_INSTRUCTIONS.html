What is Plink?

Used to work with genotype data to perform different analysis, including; * Quality control which involves cleaning data to remove errors and problematic samples * Associations studies which find connections between genetic variants and diseases or traits * Analyse genetic differences between populations

What is command line and why does PLINK use it?

Instead of having a user-interface like a website, you need to type the instrucitons in code to make the software work. PLINK does this to; * increase efficiency and speed (not using graphics makes the functions quicker, as well as using less computer resources) * you can wrtie scripts wich can automate rpeetitive analysis * run multiple PLINK instructions together

Plink and PRS

  • Building a polygenic risk score requires PLINK because you need to prepare the relevant data and extract the relevant variants.
  • PLINK can calculate the actual PRS by summing up the effects of all relevant genetic variants for each person
  • PLINK helps filter variants based on statistical significance and applies weights (effect sizes) to each variant in the score calculation

Soft Code Set Up

This is the script which makes it easier to run PLINK. This shows how to make “shortcuts” which mean that we don’t have to type in long file names which are stored in places with lots of different files. It is like a portkey which deposits that file in the code without writing all the directions. Soft coding is when you write code which is flexible. The key components like file paths, parameters, thresholds and options are stored as variables or command line arguments. To me this is like writing a function in R to avoid writing repetitive code. Soft code is like writing shortcuts. Also, if you want soft code to run you must run it each time you want to run that code (a bit like running a library at the start in R).

The advantages of soft coding are: - Improves reproducibility - Reduces human error - Makes analyses efficient - Supports automation and rerunning analyses - Encourages good data management and documentation practices - Makes it easier to adapt analyses to new data or research questions

The instructions in the soft code defines all the paths and data sets needed for the PLINK tutorial in a flexible way. Soft coding means we define variables for all paths instead of hardcoding them in commands, making the code easier to maintain, portable, and reproducible. It also saves us a lot of time constantly writing long file names / paths

Using PLINK

PLINK Basics

  1. When using PLINK the first thing you must do is call the software, which was saved as an app in the folder earlier.
  2. The next thing you do is import the data file. One of these options.
  • bfile prefix loads binary files (.bed, .bim, .fam)
  • ped file.ped and –map file.map loads text based PED/MAP files
  • vcf file.vcf loads files in a VCE format.
  1. Next you apply flags to the data which are operations or filters you want to perform on the data. For example;
  • –chr 1 keeps only chromosome 1
  • –maf 0.05 removes SNPs with minor allele frequency below 5%
  • –recode converts the format
  • –hardy calculates Hardy-Weinberg equilibrium
  1. Then to set the output file name. This –out flag specifies where and what to name the results.
  • –out results/mydata creates output files like mydata.bed, mydata.log, etc.

Types of Files

As the genotype data got bigger the filing system needed to change. The .ped file shows the family information, the ID number, sometimes there will be PATID and MOTID which show the parental ID. If there is a phenotype column the values will usually be -9, because there is rarely phenotype data available. The .bed file shows the SNP data. The .bim file shows the information about the SNP like the position on the chromosome and the identifier.

Genotype files (not advisable with whole genome data ) Compressed genotype files (standard format)
.ped – pedigree file (with genotype information) .bim – binary map file
.map – genotype map file .bed – binary ped file (with genotype information)
.fam – binary fam file

.bed files

  • Primary representation of genotype calls at biallelic variants. Must be accompanied by .bim and .fam files.
  • Loaded with –bfile; generated in many situations, most notably when the –make-bed command is used.
  • The two-bit genotype codes have the following meanings:
    • 00 Homozygous for first allele in .bim file
    • 01 Missing genotype
    • 10 Heterozygous
    • 11 Homozygous for second allele in .bim file

.ped

  • Original standard text format for sample pedigree information and genotype calls. Normally must be accompanied by a .map file;
  • A text file with a header line, and then one line per variant with the following 4-7 fields:
    • CHR Chromosome code
    • SNP Variant identifier
    • BETA Regression slope for real data. Only present with “–qfam emp-se”.
    • EMP_BETA Sample mean of permutation regression slopes. Only present with “–qfam emp-se”.
    • EMP_SE Sample stdev of permutation regression slopes. Only present with “–qfam emp-se”.
    • EMP1 Empirical p-value (pointwise), or lower-p-value permutation count
    • NP Number of permutations performed for this variant

.fam

  • Sample information file accompanying a .bed binary genotype table. Must be accompanied by .bed and .bim files.
  • Loaded with –bfile; generated when the –make-bed command is used. A text file with no header line, and one line per sample with the following six fields:

FID — Family ID IID — Individual ID PAT — Paternal ID (0 if unknown) MAT — Maternal ID (0 if unknown) SEX — Sex code (1 = male, 2 = female, 0 = unknown) PHENO — Phenotype value (1 = control, 2 = case, -9/0 = missing)

.map

  • Variant information file accompanying a .ped text genotype table. Normally must be accompanied by a .ped file.
  • A text file with no header line, and one line per variant with the following 3-4 fields:

CHR — Chromosome code SNP — Variant identifier CM — Position in morgans or centimorgans (safe to use dummy value of ‘0’) BP — Base-pair coordinate (1-based)

Data Import and Export

Depending on the file type you are using, you use a different command to import into plink. * –bfile > import plink binary files bim bed fam if they have a common name * –bim –bed –fam > import bim bed fam allowing for different names * –file > import ped and map files if they have a common name * –ped –map > import ped and map allowing for different names

Depending on the file type you are using, you use a different command to export from PLINK * –make-bed > export as PLINK binary files bim bed fam * –recode > export as PLINK files ped map

Filtering Markers

Why?

  • To exclude markers which fail the quality control
  • To extract markers which have common characteristics (e.g. overlap with gene sets). Overlap with gene sets, is where you only keep the variants which fall within specific genes or regulatory regions you’re interested in. Or variants in coding regions, variants associated with a particular trait, variants with specific allele frequencies, etc.
  • If you’re studying a particular disease or pathway, you might only care about variants in genes related to that disease.
  • Working with fewer variants speeds up analysis and reduces memory requirements.
  • Results are easier to interpret when focused on a relevant subset rather than genome-wide noise

How?

  • We need a text file with the list of MarkerID to exclude or extract.
  • PLINK used regular expression tools (so MarkerID are case sensitive)

Limiting to common markers

We often need to work with a common set of markers to remove potential missing bias and consolidate datasets to overlap. We can use a small routine including

  • Awk to select SNPs
  • Sort to sort SNPs
  • Comm to identify the common SNPs
  • comm -1 = Suppress unique to file1. or comm -2 = Suppress unique to file2. or comm -12 comm -3 = Suppress common to file1 and file2
  • Keep to extract the common SNPs

Use awk to print the marker ID column 2 from the bim files. Sort the output using the sort pipe and print to snp-list file.

awk '{print $2}' "$TMP1/HM3_chr22_ceu.bim" | sort > "$TMP1/HM3_chr_22_ceu.snps"
awk '{print $2}' "$TMP1/KG3_chr22_gbr.bim" | sort > "$TMP1/KG3_chr22_gbr.snps"

This is preparing data for comparing two datasets (likely HapMap3 and 1000 Genomes data for chromosome 22). By extracting and sorting the SNP IDs into separate files, you can then compare which SNPs are shared between the two datasets or which are unique to each one—useful for data merging or quality control in genetic studies.

  1. awk ‘{print 2}’ “TMP1/HM3_chr22_ceu.bim” reads the file HM3_chr22_ceu.bim and prints the second column (field) of each line. So $2 extracts the SNP IDs.
  2. sort > “TMP1/HM3_chr_22_ceu.snps” pipes that output to sort, which arranges the SNP IDs alphabetically/numerically, then redirects the sorted result to a new file called HM3_chr_22_ceu.snps.

Use the comm to identify the common SNPs in the sorted snp list file

comm -12 \
"TMP1/HM3_chr22_ceu.snps" "$TMP1/KG3_chr22_gbr.snps" \
> "$TMP1/common_chr22.extract"

This command finds the SNPs that are present in both files. 1. comm (short for “common”) compares two sorted files line by line and outputs three columns: • Column 1: lines only in the first file • Column 2: lines only in the second file • Column 3: lines in both files

The flags -12 suppress columns 1 and 2, leaving only column 3—the lines that appear in both files. Compares the two SNP lists you created earlier. Extracts only the SNP IDs that exist in both HM3_chr22_ceu.snps AND KG3_chr22_gbr.snps. Saves those common SNPs to common_chr22.extract

This gives you a list of SNPs shared between the HapMap3 and 1000 Genomes datasets. You’d typically use this file next with other genomic tools (like PLINK) to extract just these common variants for downstream analysis—ensuring you’re only working with SNPs that are present in both datasets, which is important for merging or comparing data across studies.

Use PLINK and the –extract flag to extract the common SNPs and create new PLINK binaries

"$PLINK" --bfile "$TMP1/HM3_chr22_ceu" --extract "$TMP1/common_chr22.extract" --make-bed \
--out "$TMP1/HM3_chr22_common_ceu"
"$PLINK" --bfile "$TMP1/KG3_chr22_gbr" --extract "$TMP1/common_chr22.extract" --make-bed \
--out "$TMP1/KG3_chr22_common_gbr"

This command uses PLINK to filter the HapMap3 dataset to include only the common SNPs. 1. “PLINK” runs the PLINK executable stored in the variable \(PLINK. 2. --bfile "\)TMP1/HM3_chr22_ceu” specifies the input binary file set. 3. –extract “\(TMP1/common_chr22.extract" tells PLINK to keep only the SNPs listed in your common_chr22.extract file (the SNPs shared between both datasets). Filtering Data – Linkage Disequilibrium

What is disequilibrium? - a loss or lack of equilibrium or stability, especially in relation to supply, demand, and prices.

What is linkage disequilibrium? - the nonrandom association of alleles at different loci — is a sensitive indicator of the population genetic forces that structure a genome. LD provides information about past events and it constrains the potential response to both natural and artificial selection. LD throughout the genome reflects the population history, the breeding system and the pattern of geographic subdivision, whereas LD in each genomic region reflects the history of natural selection, gene conversion, mutation and other forces that cause gene-frequency evolution.

Pretend that genes are friends. Friends like to hang out and go to the same places with each other. LD is a metric that describes how often you expect to find these friends hanging out in the same place. High LD means you find the friends hanging out more often. Lower LD means you find them hanging out less often. When D = 0, it means the two genes aren’t actually friends and we don’t expect to find them hanging out other than, perhaps, a random party. Similarly, if two people [genes] live in two different cities [species], we do not expect to find them hanging out with each other in the same place. This is simply because they both live in different cities [species] and do not travel.

Clumping and Pruning

We can prune using LD to reduce data footprint. Tagging was motivated by reducing the data cost of experiments. Pruning is motivated by reducing the data in experiments. Some approaches require “independence” in the sample. This means we can reduce an “already genotyped” sample containgin independent markers. Also reduced computational burden.

  1. Start with All Markers
  2. Calculate the LD between Marker 1 and Marker N Often limited to a window of 1000 Kb
  3. Remove any Marker with and LD r2 > threshold 0.9 = high threshold only drop strong proxy 0.2 = low threshold and drop all markers showing some LD i.e. drops more SNPs
  4. Move to next Marker in the list (note Marker 2 may have been pruned out)
  5. Repeat steps 2 – 5 for all markers in dataset
  6. Report remaining list of SNP = LD-independent of one another

LD pruning

Removes redundant genetic markers. In the genome, nearby SNPs are often inherited together (they’re in linkage disequilibrium). If two SNPs are highly correlated, they contain essentially the same genetic information—analyzing both is wasteful. Pruning keeps only a subset of independent markers that capture the same information.

We can limit our dataset using the –indep-pairwise flag in PLINK. We need to include three parameters.

  • The window in Kb to ignore if greater than size
  • The number of SNPs to examine in each pass
  • R2 threshold to consider as being in linkage disequilibrium

To calculate LD between single maker and define a set of independent markers.

"$PLINK" --bfile "$HM3" --indep-pairwise 1000 10 0.2 --out "$TMP1/HM3_pruned"
  1. –bfile “HM3” loads your HapMap3 genotype data.
  2. –indep-pairwise 1000 10 0.2 is the key flag that performs LD-based pruning with three parameters:
  1. 1000 is the window size in SNPs. PLINK examines 1000 SNPs at a time in a sliding window across the genome.
  2. 10 is the step size. After analyzing the first 1000 SNPs, PLINK shifts the window forward by 10 SNPs and repeats. This creates overlapping windows so no markers are missed.
  3. 0.2 is the LD threshold (r² value). SNPs with pairwise correlation higher than 0.2 are considered in LD with each other. PLINK removes one SNP from each pair that exceeds this threshold.
  1. –out “$TMP1/HM3_pruned” specifies the output file prefix.

PLINK slides through your genome in overlapping windows, calculating the correlation (LD) between all pairs of SNPs. Whenever two SNPs have r² > 0.2, it marks one for removal. The result is a set of SNPs that are largely independent of each other—no two remaining SNPs are in strong LD.

Clumping

  • Indep-pairwise select SNPs in order on the chromosome – which is good for thinning data.
  • –clump – prune from a list of SNPs, good for thinning association data and prioritises top association SNPs based on P-value
  • In –indep-pairwise the order of the SNPs is determined by position on the chromosome
  • In –clump the order of SNPs is determined by the association P-value.

Instructions

  1. Start with All Markers sort based on association p-value
  2. Define your reporting thresholds (based on association p-value)
  1. threshold 1 = list of markers to be reported (index SNPs)
  2. threshold 2 = list of marker to be compared for LD
  1. Limit the list to all markers that are associated at threshold 2
  2. Calculate the LD between the top associated Marker (Marker 1) and Marker N
  1. often limited to a window of 1000 Kb
  1. Remove any Marker with and LD r2 > threshold
  2. Move to next Marker in the P-value list
  3. Repeat steps 4 – 7 until all markers that meet threshold 1 have been analysed (or pruned)
  4. Report remaining list of SNP = LD-independent of one another
